# 增量升级与授权持久化说明

## 本次改动

- 关于入口改为带文字的“关于”按钮，并保持在面板右侧；关于页的“了解一下”和“检查更新”动作均显式右对齐。
- 接入 GitHub Release 页内更新：用户可从“关于 → 检查更新”直接检查、下载、解压与安装，不会跳转浏览器。
- 更新包在替换应用前会校验 bundle identifier、macOS Code Signing 签名；正式发布时还会比对 Team ID，避免错误来源的更新包被安装。
- 发布脚本不再允许使用 ad-hoc 签名生成可自动更新的正式包：正式发布必须使用固定的 `Developer ID Application` 身份，避免每次发布改变 macOS 识别的应用身份。

## 为什么此前会重新询问密码/授权

应用源代码没有 Keychain 读写逻辑。当前旧发布脚本采用 ad-hoc (`codesign --sign -`) 重签名并通过 ZIP 手工覆盖安装。ad-hoc 签名的 designated requirement 与具体构建绑定，macOS 无法可靠地将新版识别为已授权的同一个应用，因此辅助功能、钥匙链访问控制或 Gatekeeper 可能再次提示。

macOS 不允许应用静默预授权钥匙链或隐私权限；正确做法是让每个后续版本维持相同的 Bundle ID 和受信任的 Developer ID 签名身份。首次由用户授权后，后续通过 Sparkle 原子更新的版本才有机会继续满足同一代码身份要求。

## 发布前必须完成的一次性配置

1. 配置 Apple Developer Program 的 `Developer ID Application` 证书，并确认它在发布机钥匙链中可用。
2. 使用同一 Developer ID 对应用及每次更新签名，并完成 Apple 公证；将 ZIP 作为 GitHub Release 附件上传。
3. 首次从旧 ad-hoc ZIP 迁移到 Developer ID 版本时，macOS 可能仍会要求一次确认。这是代码身份切换造成的，之后不应再以卸载/重装方式升级。

## 发布命令

```bash
export DEVELOPER_ID_APPLICATION='Developer ID Application: <团队名称> (<TEAM_ID>)'
./script/package_release.sh
```

> 公证及 GitHub Release 上传需要在发布环境/CI 中完成；本地脚本只负责生成已签名应用和 ZIP。上线前请用一台已安装旧版的测试机执行“检查更新”，确认辅助功能和其它既有授权没有被重置。
